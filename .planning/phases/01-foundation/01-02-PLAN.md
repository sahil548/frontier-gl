  <name>Task 2: Implement entity CRUD API route handlers</name>
  <files>

  <action>
    1. Create `src/app/api/entities/route.ts` with GET and POST handlers:

       **GET /api/entities:**
       - Call `auth()` from @clerk/nextjs/server -- return 401 if no userId
       - Look up the internal User record by clerkId. If not found, create one (upsert pattern: first API call after Clerk signup creates the User row)
       - Query `prisma.entity.findMany` where createdById = internal user id AND isActive = true, ordered by name asc
       - Serialize entities and return via successResponse

       **POST /api/entities:**
       - Call `auth()` -- return 401 if no userId
       - Parse request body with `createEntitySchema.safeParse(body)`
       - If validation fails, return errorResponse with 400 and Zod errors
       - Upsert the User record by clerkId (same pattern as GET)
       - Create entity with `prisma.entity.create({data: {...result.data, createdById: userId}})`
       - Return serialized entity via successResponse with 201

       User upsert pattern: Since Clerk manages auth and we don't have a webhook sync, the first API call from a new user should create their User record. Use prisma.user.upsert with clerkId as the unique field. Get email from Clerk's auth() or currentUser().

    2. Create `src/app/api/entities/[entityId]/route.ts` with GET and PUT handlers:

       **GET /api/entities/:entityId:**
       - Auth check
       - Find entity by id, verify it belongs to the authenticated user (createdById check)
       - Return 404 if not found or not owned
       - Return serialized entity

       **PUT /api/entities/:entityId:**
       - Auth check
       - Parse body with `updateEntitySchema.safeParse(body)`
       - If validation fails, return errorResponse with 400
       - Verify entity exists and belongs to user
       - Update entity with prisma.entity.update
       - Return serialized updated entity

       For deactivation: PUT with `{isActive: false}`. The updateEntitySchema allows isActive as an optional boolean field.

    Note on API-01: These routes are at /api/entities/ and /api/entities/:entityId, not under /api/entities/:entityId/ scoping. Entity management itself is user-scoped, not entity-scoped. The /api/entities/:entityId/ scoping pattern applies to financial data WITHIN an entity (accounts, journal entries) which comes in Phase 2. Document this distinction in a code comment.
