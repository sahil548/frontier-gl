  <files>src/app/api/entities/[entityId]/bank-transactions/route.ts, src/app/api/entities/[entityId]/bank-transactions/[transactionId]/route.ts, src/app/api/entities/[entityId]/bank-transactions/bulk-categorize/route.ts</files>
  <action>
  Create three API route files following the established entity-scoped pattern (see existing routes like /api/entities/[entityId]/budgets/route.ts).

  **GET /api/entities/[entityId]/bank-transactions/route.ts:**
  - Query params: status (optional filter: PENDING | CATEGORIZED | POSTED | EXCLUDED), subledgerItemId (optional filter for per-bank-account view), page (default 1), limit (default 50)
  - Returns paginated BankTransaction list with included account (name, number) and rule (pattern) relations
  - Order by date DESC, then createdAt DESC
  - Entity ownership verification via findOwnedEntity pattern

  **POST /api/entities/[entityId]/bank-transactions/route.ts (CSV import):**
  - Validate body with csvImportSchema: { csv: string, subledgerItemId: string }
  - Verify subledgerItemId belongs to this entity and is a BANK_ACCOUNT type
  - Call parseBankStatementCsv(csv) to get parsed rows
  - For each row, generate hash via generateTransactionHash(date, amount, description, rowIndex for same-hash disambiguation)
  - Call findDuplicates to get existing hashes
  - Skip duplicates, create BankTransaction records for new ones with source: 'CSV', externalId: hash
  - Auto-apply categorization rules (fetch entity's active rules, call matchRule for each new transaction, set accountId/ruleId/status to CATEGORIZED if matched)
  - Return { imported: number, skipped: number, categorized: number, errors: string[] } following the existing budget import response pattern

  **PATCH /api/entities/[entityId]/bank-transactions/[transactionId]/route.ts:**
  - Categorize a single transaction: { accountId: string, dimensionTags?: Record<string, string> }
  - Update transaction accountId, status to CATEGORIZED
  - Return updated transaction

  **POST /api/entities/[entityId]/bank-transactions/[transactionId]/route.ts:**
  - Create JE from transaction: { postImmediately?: boolean, splits?: SplitLine[] }
  - If splits provided, validate with splitTransactionSchema (sum must equal transaction amount)
  - Look up the bank account's GL accountId from subledgerItem.accountId (this is the credit/bank side of the JE)
  - Use createJournalEntryFromTransaction to build the JE data
  - If postImmediately: create JE as POSTED, update account balances via postJournalEntry. If not: create as DRAFT
  - Update transaction status to POSTED, link journalEntryId
  - Return the created JE

  **POST /api/entities/[entityId]/bank-transactions/bulk-categorize/route.ts:**
  - Validate body with bulkCategorizeSchema: { transactionIds: string[], accountId: string, postImmediately?: boolean }
  - For each transaction: assign accountId, create JE (if postImmediately), update status
  - Wrap in a Prisma transaction for atomicity
  - Return { processed: number, journalEntries: number, errors: string[] }

  All routes: use entity ownership check, Zod validation, try/catch with errorResponse, inline serialization helpers for Decimal fields.
