  - Order by priority ASC (lower = higher priority), then createdAt ASC
  - Return serialized rules (Decimal fields to string)

  **POST /api/entities/[entityId]/bank-transactions/rules/route.ts:**
  - Validate body with categorizationRuleSchema
  - Verify accountId belongs to this entity
  - If dimensionTags provided, verify each dimensionId/tagId exists for this entity
  - Auto-assign priority: count existing active rules, new rule gets count as priority (appended at end)
  - Create CategorizationRule with entityId
  - Optionally: apply this new rule to existing PENDING transactions for this entity (find unmatched transactions whose description matches the new pattern, update their accountId and status to CATEGORIZED). Return count of retroactively matched transactions
  - Return created rule with { id, pattern, account, matchedCount }

  **PATCH /api/entities/[entityId]/bank-transactions/rules/[ruleId]/route.ts:**
  - Validate body with partial categorizationRuleSchema (all fields optional)
  - Verify rule belongs to this entity
  - Update rule fields
  - Return updated rule

  **DELETE /api/entities/[entityId]/bank-transactions/rules/[ruleId]/route.ts:**
  - Soft delete: set isActive = false (matching dimension soft-delete pattern from Phase 6)
  - Do NOT remove accountId from already-categorized transactions (those keep their categorization)
  - Return { success: true }

  All routes follow existing patterns: Clerk auth, entity ownership check, Zod validation, try/catch with errorResponse, serialization for Decimal fields.
