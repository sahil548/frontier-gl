model Account {
  id          String      @id @default(cuid())
  entityId    String
  number      String
  name        String

  <action>
    1. Create `src/app/api/entities/[entityId]/accounts/route.ts` with GET and POST:

       **GET /api/entities/:entityId/accounts:**
       - Auth check via auth() from @clerk/nextjs/server
       - Verify entity belongs to authenticated user
       - Query accounts where entityId matches, include children and balance relation
       - For "all" entityId: return accounts across all user's entities (use entityScope helper)
       - Order by: number ASC (so parent accounts appear before children naturally)
       - Include balance relation (AccountBalance) for inline display
       - Serialize Decimal fields to strings before returning
       - Return flat array -- the client handles indentation based on parentId

       **POST /api/entities/:entityId/accounts:**
       - Auth check
       - Validate body with createAccountSchema.safeParse
       - If parentId provided: verify parent exists, belongs to same entity, and has no parent itself (2-level max -- reject if parent already has a parentId)
       - If parentId provided: verify parent type matches the new account type (sub-accounts inherit parent type)
       - CRITICAL: Reject if account would be a 3rd level (parent.parentId is not null)
       - Create account with prisma.account.create
       - Create empty AccountBalance row for the new account (debitTotal=0, creditTotal=0, balance=0)
       - Return serialized account with 201

    2. Create `src/app/api/entities/[entityId]/accounts/[accountId]/route.ts` with GET and PUT:

       **GET:** Return single account with balance and children
       **PUT:** Validate with updateAccountSchema, update account. If isActive=false (deactivation), verify account has no children that are still active. Deactivated accounts keep their data but are hidden from selectors.

    3. Create `src/app/api/entities/[entityId]/accounts/template/route.ts` with POST:

       **POST /api/entities/:entityId/accounts/template:**
       - Auth check, verify entity ownership
       - Call applyTemplate(entityId) from src/lib/accounts/template.ts
       - Return { inserted, skipped } counts
       - This endpoint applies the Family Office Standard template to the entity

    4. Create `src/app/api/entities/[entityId]/accounts/next-number/route.ts` with GET:

       **GET /api/entities/:entityId/accounts/next-number?parentNumber=10000:**
       - Auth check
       - Call suggestNextAccountNumber(entityId, parentNumber) from src/lib/accounts/next-number.ts
       - Return { suggestedNumber: "10100" }

    Note on parent account balance: When fetching accounts, parent account balance is computed by summing children's AccountBalance records at query time. Do NOT store parent balance separately -- compute it in the GET handler by aggregating children. This avoids cascading trigger complexity per 02-RESEARCH.md Pitfall 4.
