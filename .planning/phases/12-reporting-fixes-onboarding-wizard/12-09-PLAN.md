<objective>
Close UAT Test 6 (major): the bank CSV import is currently single-account-scoped (`subledgerItemId` required in the POST body; `bank-feed/page.tsx:231-234` blocks import until a bank account is selected). UAT user wants multi-account CSVs where the Account column is just another mappable dimension.

Architectural change (but minimal surface):
1. Add `account` as an OPTIONAL role in the bank Column Mapping UI.
2. When the user maps the Account column, `bank-feed/page.tsx` switches from "single subledgerItem" mode to "per-row account resolution" mode.
3. The API accepts either the legacy `subledgerItemId` string OR a new `accountResolution: { strategy: "per-row"; column: string; matchBy: "name" | "number" }` descriptor.
4. The CSV parser returns the per-row account reference alongside date/description/amount. The API resolves each reference to a `subledgerItem.id` by matching against the entity's `subledgerItem.name` OR the linked `account.number`; unresolved rows are returned in the error list (not imported).
5. `parseBankStatementCsv` is extended so parsed rows include `accountRef?: string` when mapping.account is present.

This is deliberately a checkpoint plan — a major data-routing change warrants human smoke-test before considering gap closed.

Purpose: Treat accounts/holdings as mappable dimensions (parity with date, amount, description) per UAT user's explicit request.

- Route changes resolving account per row
- Column Mapping UI adds 'Account' role for bank
- Bank Feed page UX: selector becomes optional, a clear "Multi-account from CSV" state indicator
- Checkpoint for user to verify in Chrome
