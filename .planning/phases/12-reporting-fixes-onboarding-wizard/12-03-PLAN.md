  <action>
1. **Computation utility** -- Create `src/lib/budgets/rate-based.ts`:
   ```typescript
   import Decimal from "decimal.js";

   interface RateBudgetInput {
     holdingMarketValue: Decimal;
     annualReturnRate: number; // e.g., 0.08 for 8%
   }

   export function computeMonthlyBudget(input: RateBudgetInput): Decimal {
     return input.holdingMarketValue
       .times(new Decimal(input.annualReturnRate))
       .dividedBy(12)
       .toDecimalPlaces(4);
   }
   ```
   Import Decimal from "decimal.js" (already installed, used throughout project per DI-01).

2. **API endpoint** -- Create `src/app/api/entities/[entityId]/budgets/rate-target/route.ts`:
   - POST handler accepting: `{ holdingId: string, accountId: string, annualRate: number, fiscalYear: number }`
   - Validate with Zod: holdingId is cuid, accountId is cuid, annualRate is number between 0 and 1, fiscalYear is positive integer
   - Auth + entity access check (standard pattern)
   - Fetch the holding (SubledgerItem) by holdingId -- get its `marketValue`
   - If marketValue is null, return 400 "Holding has no market value"
   - Call `computeMonthlyBudget({ holdingMarketValue: marketValue, annualReturnRate: annualRate })`
   - Generate 12 budget lines (one per month of the fiscal year) using the entity's fiscal year end to determine month boundaries
   - Upsert each budget line using `prisma.budget.upsert()` with `@@unique([entityId, accountId, month])` constraint
   - Return 200 with `{ created: 12, monthlyAmount: computedAmount.toString() }`

3. **Fill in test stubs**:

   - `src/__tests__/api/rate-budget.test.ts`: Test the validation logic (mock the computation). Test that the endpoint rejects negative rates and missing holding values.
