       - Import prisma, HOLDING_TYPE_TO_GL, DEFAULT_POSITION_NAME, createPositionGLAccount, createHoldingSummaryAccount
       - Main function `migrateHoldingsToPositionModel()`:
         a. Fetch all active SubledgerItems with their account and positions (include: { account: true, positions: { where: { isActive: true } } })
         b. Filter to only holdings where accountId still points to a non-summary account (skip already-migrated holdings)
            Detection: if holding has any positions with accountId set, it's already migrated
         c. For each unmigrated holding, within a $transaction:
            i. Look up glMapping from HOLDING_TYPE_TO_GL[holding.itemType]
            ii. Call createHoldingSummaryAccount(tx, holding.entityId, glMapping.parentPrefix, holding.name, glMapping.accountType) to get the new summary account
            iii. If holding has NO positions:
                 - Re-parent the holding's existing GL account: update account.parentId to summaryAccount.id
                 - Create a default Position with name from DEFAULT_POSITION_NAME, accountId set to the holding's existing GL account (the one that was re-parented)
                 - Determine positionType using the same mapping as Task 2 of Plan 01 (BANK_ACCOUNT -> CASH, etc.)
            iv. If holding HAS positions:
                 - For each position: call createPositionGLAccount(tx, holding.entityId, summaryAccount.id, position.name, glMapping.accountType)
                 - Update each position's accountId to the newly created leaf account
            v. Update the SubledgerItem's accountId to the new summary account
         d. Log progress: "Migrated N holdings (X with existing positions, Y with default positions created)"
       - Script is idempotent: re-running skips already-migrated holdings
       - Execute via: `npx tsx scripts/migrate-holdings-positions.ts`
       - Add a dry-run flag (--dry-run) that logs what would happen without writing
